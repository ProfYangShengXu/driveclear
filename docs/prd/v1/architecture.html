1. 架构总览

核心设计原则： 每个模块独立可测、接口统一、传统图像处理方法（非机器学习）。 所有处理模块的输入输出均为 (H, W, 3) uint8 BGR ndarray。

1.2 为什么这么分（模块划分理由）

  1. M1 / M2 与 M5 分离 — M1/M2 是"处理"模块，M5 是"感知"模块。感知逻辑不耦合在管线里，可独立测试和调参。如果要换一种退化检测策略（比如用局部方差替代暗通道均值），只需要改 M5，不动 M1。
  2. M3 / M4 独立为模块 — 因为眩光和低光的处理策略完全不同（一个是找亮区压下去，一个是找暗区提上来），合在一个文件里会违反单一职责。分开后分别测试。
  3. M6 单独作为编排层 — 把"调用哪个模块"的决策逻辑从处理逻辑中分离。这样加一个新的退化类型（比如雨纹去除）时，只需要新建模块 + 在 M5 加一个 score，不需要改编排器核心。
  4. M7 / M8 / M9 独立于应用 — 评估和消融不是产品功能，是论文/课程交付所需的实验框架。把它们放在 scripts/ 而不是 algorithms/，避免产品代码膨胀。

2. 数据流

2.1 单帧处理管线

输入帧 (H,W,3) uint8 BGR │ ├── M5: 退化检测器 │ haze_score = mean(dark_channel) × (1 - mean(saturation)) │ glare_score = ratio(bright_channel > 220) × peak_entropy │ low_light_score = sigmoid((40 - mean_luminance) / 20) │ ├── M3: 眩光抑制 (if glare_score > threshold 或 auto_detect=False) │ detect_glare → suppress_glare( + temporal_inpaint ) │ ├── M4: 低光照增强 (if low_light_score > threshold 或 auto_detect=False) │ Retinex分解 → illum_gamma + reflect_clahe → 合并 │ ├── M1: 时域雾层估计 + DCP 去雾 (if haze_score > threshold 或 auto_detect=False) │ TemporalFogEstimator.update() → dehaze() │ ├── M2: 自适应融合 + 亮度增强 (if enable_fusion/enable_enhance) │ compute_motion_map → fuse_frames → CLAHE + Sharpen + Gamma → color_restore │ └── 输出增强帧
关键设计： 当 auto_detect=True 时，M5 的 score 决定是否激活 M3/M4/M1。 当 auto_detect=False 时，用户通过 PipelineConfig 明确启用/禁用每个模块。 处理顺序固定为 眩光→低光→去雾→融合增强，因为：
① 眩光抑制需要在"原始"亮区上检测
② 低光增强在去雾之前做，避免 DCP 将暗区噪声放大

2.2 视频级数据流

用户上传视频 │ POST /api/upload │ └── video_service.save_upload() │ POST /api/process/{id} (含 PipelineConfig JSON) │ └── processing_service.process_pipeline() │ ├── 逐帧: read_frames() 生成器 → [处理管线] → write_video() │ └── 完成后 status=COMPLETED 前端轮询 GET /api/status/{id} → 进度更新 完成后 GET /api/download/{id} → 下载 DELETE /api/cleanup/{id} → 清理

4. 各模块详细设计

4.1 M3 — 眩光抑制 🆕

文件： backend/algorithms/glare.py

为什么要用传统方法： 眩光在图像中表现为高亮度、低纹理、边缘模糊的圆形/椭圆形区域。不需要神经网络——亮通道阈值 + 形态学分析 + 时域插值即可有效抑制。

算法步骤

  1. 亮通道提取： bright = max(R, G, B) → 阈值分割（>220）→ 候选眩光区域
  2. 几何过滤： 计算每个连通区域的 面积 / (π × 半径²) 圆度比 → 过滤非圆形区域（车灯保留）
  3. 局部亮度抑制： 眩光区域内像素值 = lerp(原值, 局部背景均值, strength)
  4. 时域插值： 利用前后帧对应位置的非眩光像素加权平均补全纹理

接口签名

def detect_glare(frame: np.ndarray) -> tuple[np.ndarray, float]:
    """返回 (glare_mask, glare_intensity)"""

def suppress_glare(frame: np.ndarray, glare_mask: np.ndarray,
                   prev_frame: np.ndarray | None = None,
                   strength: float = 0.7) -> np.ndarray:
    """返回抑制后的帧"""

风险措施

  • 车灯误抑制： 仅处理面积 > 300px² 且圆度 > 0.7 的区域。小光源（远处车灯）面积小，放过。
  • 无眩光帧： detect_glare 返回全零 mask → suppress_glare 直接返回原帧（M3-A3 验证）

4.2 M4 — 低光照增强 🆕

文件： backend/algorithms/low_light.py

为什么不直接用现成的 CLAHE： 现有 Phase4 的 CLAHE 是作为"后处理"做的，没有显式的照明/反射分离。对于极暗帧（mean luminance < 30），直接 CLAHE 会放大噪声。Retinex 分解可分别控制照明层（全局亮度）和反射层（纹理细节）。

算法步骤

  1. Retinex 分解（照明-反射分离）： 对亮度通道做高斯滤波（大核 σ=30）得到照明层 I = G * L，反射层 R = L / I（逐元素除法，ε=1e-6 防除零）
  2. 照明层增强： 自适应伽马 gamma = 1 + 0.6 × (1 - mean(I)) → 低亮度区域提升更多
  3. 反射层 CLAHE： clipLimit=1.5（弱于 M2 的 2.0，避免过度增强噪声）
  4. 噪声抑制： 对反射层做边缘保持滤波（双边滤波）→ 平滑噪声但保留边缘
  5. 合并： L_out = I_enhanced × R_denoised → 转回 BGR

与 M2 Phase4 的关系

如果 enable_low_light=True 且 enable_enhance=True：M4 做基础亮度提升，M2 做最终 CLAHE + 锐化。两者不冲突——M4 负责"把暗区拉起来"，M2 负责"让画面更锐利"。但如果 M4 已做 CLAHE，M2 可适当降低 clipLimit。

4.3 M5 — 退化检测器 🆕

文件： backend/algorithms/degradation_detector.py

为什么不需要 ML： 三种退化在统计层面有明确的信号特征。雾→暗通道均值高且饱和度低；眩光→亮度直方图右尾长且有高亮连通域；低光→亮度直方图集中在低端。这些都可以用 OpenCV 直方图 + 阈值做。

各 score 的计算方式

Score计算方法范围
haze_score mean(dark_channel(frame)) × (1 - mean(saturation(frame))) → sigmoid 归一化 [0, 1]
glare_score 亮通道(>220)像素比例 × top 0.1% 亮度的熵平整度 → 大且平的高亮区得分高 [0, 1]
low_light_score sigmoid((40 - mean_luminance) / 20) → 均值 40 时约 0.5，越低越高 [0, 1]

阈值策略

默认阈值：haze=0.3, glare=0.4, low_light=0.35。这些是经验值，通过 M7 评估后可在消融实验中调优。

4.4 M6 — 管线编排器 🔄 重构

7. 不做的方案（设计时已排除）

方案排除理由
深度学习端到端去雾/增强 课程为数字图像处理，要求传统方法；且 DL 模型需 GPU 训练
光流法多帧对齐 光流计算量大，且 OpenCV 的 Farneback 光流对噪声敏感。改用帧差 + 运动蒙版
频域去雾（同态滤波） 实验中发现对雾霾效果不如 DCP，且参数难调
直方图匹配色彩校正 现有 restore_natural_colors 已足够，引入额外复杂度
实时 30 FPS 处理 当前逐帧 CPU 处理 5 FPS 是合理目标，实时需要 GPU/ffmpeg 硬件加速
